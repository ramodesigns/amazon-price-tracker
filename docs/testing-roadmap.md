## Testing tiers: unit vs. component vs. integration

Three tiers now, split by what's real vs. faked and where they live:

| Tier | What's real | What's faked | Directory | CI |
|---|---|---|---|---|
| Unit | The class under test | Everything else - no REST dispatch, no DB writes | `tests/unit/` | Always |
| Component | REST dispatch, DB, WP core | Amazon PA-API (via `pre_http_request`) | `tests/component/` | Always |
| Integration | REST dispatch, DB, WP core, Amazon PA-API | Nothing | `tests/integration/` | Opt-in only (env-var gated) |

**Determination - which tier a given test belongs in:**

- Testing *this plugin's own logic* (routing, permission checks, validation, error mapping, DB row shape, response shape) → **component**. Amazon's actual response content is irrelevant to what's being verified, and a canned fixture is arguably better than the real thing here: it lets you deterministically hit every branch (success, ASIN not found, throttled, malformed response, partial data) on demand, which live Amazon won't reliably reproduce.
- Testing that the plugin's real request/response code actually *interoperates* with Amazon's real API (correct signing, correct per-region hosts/paths, real payload shape, credential handling) → **integration**. This is exactly what caught the `/paapi5/GetItems` path-casing bug tonight - no amount of mocking would have found that, since a hand-written mock would just have encoded the same wrong assumption the production code had.
- Rule of thumb: component tests should heavily outnumber integration tests. Integration tests answer "does the wire format actually work," not "does every branch of business logic work" - exercising every branch against a live, rate-limited, credentialed external service is slow and expensive for no extra signal.

**Mechanics for component tests:** intercept at the `pre_http_request` filter, which `wp_remote_post()` already checks before making a real socket call. A canned Amazon-shaped JSON response comes back through the *same* code path that runs in production - `Amazon_API::request()` still builds the real signed request, still parses whatever comes back - so component tests still catch bugs in URL building, header construction, and response parsing (they would have caught the casing bug too, once tests existed at all), just not "does Amazon's server accept our signature."

  - **Two real bugs found and fixed by this testing pass** (both previously invisible - `get_items()`'s search and `calculate_aggregations()`'s `aggregate=daily` had zero test coverage before this):
    1. **Search was silently case-sensitive.** `get_items()`'s title/brand search did `JSON_UNQUOTE(JSON_EXTRACT(p.facts, '$.title')) LIKE %s` - but MySQL/MariaDB's JSON functions return `utf8mb4_bin` (case-sensitive) regardless of the source column's collation, so searching "wireless" would never match a product titled "Wireless Mouse". Fixed by wrapping both sides in `LOWER()` rather than hardcoding a `COLLATE` (which could mismatch an install's actual `charset_collate` and throw an "illegal mix of collations" error instead).
    2. **`?aggregate=daily` (and the `default` case, which uses the same format) silently broke.** `calculate_aggregations()` splices its `DATE_FORMAT()` pattern directly into SQL that's later run through `$wpdb->prepare()`. The `'%Y-%m-%d'` daily/default format contains a literal `%d`, which `wpdb::prepare()`'s own placeholder scanner can't distinguish from a real `%d` (integer) bind placeholder - it throws off the placeholder/argument count, `prepare()` returns `null`, and the aggregation query fatals with a `TypeError` in `array_map()`. (`weekly`/`monthly` happened to dodge this by accident - their formats contain no `d`.) Fixed by doubling every `%` in the format string (`str_replace('%', '%%', ...)`) before splicing it in, which is wpdb's documented way to pass through a literal percent sign.

- [x] `composer require --dev infection/infection`
- [x] Add `infection.json5` config, scoped to `includes/Helpers/` first (highest existing coverage, cheapest to get a clean baseline) before widening to `includes/Services/` and `includes/API/Controllers/`.
- [x] Needs a coverage driver and a passing baseline test run to seed mutation targets. Local dev machine had neither PCOV nor a running MySQL/WP test environment at the time - re-provisioned both: `brew install mariadb` (+ a dedicated `wp_test` DB user) and `bin/install-wp-tests.sh` to pull WP core/test-lib into `$TMPDIR/wordpress-tests-lib`. Xdebug (already installed) works fine as the coverage driver; PCOV was not reinstalled since Xdebug covers this size of codebase without a meaningful speed problem.
- [x] First run (2026-07-13) surfaced weak assertions exactly as expected: 208/252 mutants killed, 44 escaped (82% Covered Code MSI) across `Encryption`, `Response`, `Validation`. Strengthened `tests/unit/test-encryption.php`, `test-response.php`, `test-validation.php` with the missing edge cases (exact default status codes, ceil-vs-round pagination boundary, whitespace/case-fold boundary inputs, pagination clamped against the *default* `max_per_page`, category truncation with a non-uniform string so an off-by-one offset can't hide behind repeated characters, and a `ReflectionMethod` test pinning `Encryption::get_key()`'s exact derivation since the public encrypt/decrypt round trip is internally self-consistent regardless of how the key is derived). Result: **239/252 killed, 94% Covered Code MSI**.
- [x] Remaining 13 escapees were individually hand-verified (not assumed) to be genuine equivalent mutants - the mutated code is behaviorally identical to the original for every possible input, so no test could ever kill them without changing what "correct" means:
  - `Validation::is_valid_asin()`'s `(bool)` cast on `preg_match()` - the `: bool` return-type declaration coerces the int at the call boundary regardless (confirmed by direct execution).
  - `Validation::validate_price()`'s `(float)` cast - `round()`'s `int|float` parameter type coerces a numeric string the same way (confirmed by direct execution).
  - Two `sanitize_text_field(trim($x))` → `sanitize_text_field($x)` mutants (`validate_search_query`, `validate_category`) - WP core's `sanitize_text_field()` trims internally as its last step (confirmed by direct execution), so the outer `trim()` is redundant.
  - `validate_region_list()`'s `array_map('trim', ...)` removal - each code gets trimmed again inside `normalize_region()` right after (confirmed by direct execution).
  - `validate_pagination()`'s `$page = 1` default → `0` - `max(1, $page)` clamps either default to the same result.
  - Two `ArrayItemRemoval` mutants (`'none'` dropped from `validate_aggregation()`'s allow-list, `'created_at'` dropped from `validate_product_sort_field()`'s) - the removed value is also the function's own fallback default, so failing the allow-list check and hitting the fallback produces the same output as passing it.
  - `validate_bool()`'s early `return $value` (inside `if (is_bool($value))`) removed - execution falls through to the final `return (bool) $value`, which is a no-op cast on an already-boolean value.
  - `validate_category()`'s `strlen($category) <= 255` → `< 255` - the only input where the two operators disagree is length exactly 255, and both branches produce an identical 255-character string at that exact boundary (`substr($s, 0, 255)` on a 255-char string returns the string unchanged).
  - `Encryption::get_key()`'s `&&` → `||` on `defined('AUTH_KEY') && AUTH_KEY` - can't be distinguished without an environment where `AUTH_KEY` is defined but falsy, which isn't constructible once WP's test bootstrap has already defined the constant as a truthy string.
  - Two `ReturnRemoval` mutants in `Encryption::decrypt()` (the `empty($data)` and `base64_decode() === false` early returns) - removing them still falls through empty/garbage input to `openssl_decrypt()` failing and the function's own final `$decrypted !== false ? $decrypted : ''` fallback, landing on the same `''` output.
  - Not re-litigated: the pre-existing `Encryption::get_key()`'s no-`AUTH_KEY` fallback branch (lines 39-47) remains genuinely unreachable from this environment, as noted in [Done](#done) above.

## Real WordPress environment for manual testing

Needed before E2E automation can exist. Standard options:

- [ ] `@wordpress/env` (`wp-env`) - official WordPress tooling, Docker-based, simplest to wire up for a single plugin, config lives in `.wp-env.json`.
- [ ] Alternative: plain `docker-compose.yml` with `wordpress` + `mysql` images if you want more control than `wp-env` gives you.
- [ ] Either way: mount this repo into `wp-content/plugins/`, activate it, then `docs/api-test-checklist.md` already exists as your manual QA script - Postman collection referenced there is `docs/api-collection.json`, worth confirming that's still current.
- [ ] Real credentials for manual PA-API testing should go through the plugin's own encrypted settings storage (as designed) rather than env vars - that's the real user flow, unlike the test shortcut we used tonight.

## E2E browser automation (after the container environment exists)

- [ ] Playwright is the more natural fit over Cypress here given WordPress's traditional (non-SPA) admin UI and the REST API surface - better multi-tab/multi-context support if testing Application Password creation flows.
- [ ] Start with the golden path: create Application Password → configure PA-API credentials → create a product via REST → see it reflected wherever the plugin surfaces it in wp-admin.
- [ ] This tier only makes sense once the `wp-env`/Docker environment above is reliable and scriptable (spin up, seed, tear down) - don't start this before that's solid.

## Performance testing (lower priority)

- [ ] Once the container environment exists: `k6` or `wrk` against the REST endpoints, particularly `GET /products` (the hand-rolled SQL with `JSON_EXTRACT` filtering/sorting in `Products_Controller::get_items()` is the most likely bottleneck at scale) and `bulk_refresh` (sequential Amazon API calls with a 100ms inter-batch delay - throughput-bound by design).
- [ ] Worth a realistic data volume seed (thousands of products/price-history rows) rather than testing against an empty dev DB - `tests/seed-test-data.php` may need extending for this.
