- [ ] FEAT-07: Batch bulk create's Amazon calls (PRIORITY: MEDIUM)
      CONTEXT (discovered 2026-07-15 while designing integration tests):
      Products_Controller::bulk_create() loops create_item() per product,
      so N items cost N single-ASIN getItems calls. The Creators API
      accepts up to 10 itemIds per getItems call - Product_Service::
      bulk_refresh() already batches this way (array_chunk 10 + 100ms
      inter-batch delay), so bulk create of 100 products currently makes
      100 Amazon calls where ~10 would do. Faster for the caller, and an
      order of magnitude kinder to Amazon's rate limits.
      DESIGN CONSTRAINTS (so the refactor doesn't regress behavior):
        - Run ALL per-item pre-checks first (in-request duplicate detection,
          blacklist, already-exists conflict, and the non-admin daily rate
          limit - capping accepted items at the caller's remaining quota),
          then batch-fetch only the survivors, grouped by region like
          bulk_refresh (each region needs its own client/marketplace).
        - An ASIN absent from an otherwise-healthy batch response is a
          per-item ASIN_NOT_FOUND failure (mirror bulk_refresh's handling;
          note the Creators API has no per-item not-found error - absence
          IS the signal, see MIGR-09 notes).
        - Response shape (success_count/failure_count/results) must not
          change - docs/openapi.yaml documents it and consumers rely on it.
          Per-item error codes must survive the refactor too.
        - create_product()'s per-item transaction/rollback semantics must
          be preserved per inserted row, not wrapped around the whole batch.
      TESTS TO UPDATE WHEN THIS LANDS:
        - tests/component/test-products-controller.php bulk-create tests
          (mock consumption changes: one queued response per 10-item batch,
          not one per item).
        - tests/integration/test-bulk-create.php - its docblock explicitly
          documents the current fan-out behavior and drives bulk REFRESH
          for the batch wire shape; after this lands, bulk create becomes
          a true batch sender and the file should be reworded (and the
          batch-payload live test could then use either operation).

# PRIORITY: HIGH
# ACTION: Follow this sequence when testing in pre-prod environment
# NOTE: This is a test script to validate all functionality works

STEP 1: Plugin Activation
  - Activate plugin via WordPress admin
  - Verify: wp_apt_products table exists
  - Verify: wp_apt_price_history table exists
  - Verify: wp_apt_user_settings table exists
  - Verify: wp_apt_blacklist table exists

  - Endpoint: PUT /wp-json/amazon-price-tracker/v1/settings

  - Expected: 201 Created (first time) or 200 OK (update)

STEP 3: Validate Amazon Credentials
  - Endpoint: POST /wp-json/amazon-price-tracker/v1/settings/validate
  - Expected: {"valid": true, "message": "..."}

STEP 4: Create First Product
  - Endpoint: POST /wp-json/amazon-price-tracker/v1/products
  - Body: {"asin": "B08N5WRWNW", "region": "US"}
  - Expected: 201 Created with product object

STEP 5: Verify Product Data
  - Endpoint: GET /wp-json/amazon-price-tracker/v1/products/{id}
  - Verify: images array populated
  - Verify: facts object contains title, brand, etc.

STEP 6: Test Price Refresh
  - Endpoint: POST /wp-json/amazon-price-tracker/v1/products/{id}/refresh
  - Expected: 200 OK with updated product
  - Verify: New record in apt_price_history

STEP 7: Test Price History
  - Endpoint: GET /wp-json/amazon-price-tracker/v1/products/{id}/prices
  - Verify: Returns price records
  - Test with ?aggregate=daily

STEP 8: Test Blacklist
  - Endpoint: POST /wp-json/amazon-price-tracker/v1/blacklist
  - Body: {"asin": "B000000000", "region": "US", "reason": "Test"}
  - Verify: 201 Created
  - Try to create product with blacklisted ASIN
  - Verify: 403 Forbidden with BLACKLISTED code

STEP 9: Test Rate Limiting (as non-admin user)
  - Create standard WordPress user (Subscriber role)
  - Generate Application Password for user
  - Create 50 products successfully
  - Attempt 51st product
  - Verify: 429 Rate Limit Exceeded

STEP 10: Test Scheduled Refresh
  - Go to Settings > Amazon Price Tracker
  - Click "Run Manual Refresh"
  - Verify: Success message with count
  - Check price_history table for new records
